# 中期进度报告

## 一、项目名称

讨薪方向可解释法律 AI 系统（TaoxinAI）

## 二、项目目标

本项目当前聚焦“讨薪/拖欠劳动报酬”场景，目标不是做普通法律问答系统，而是搭建一个分阶段的法律论证系统。系统将围绕案件输入，逐步完成：

- 检索法条、案例与模板
- 识别争点
- 抽取关键事实
- 进行要件匹配
- 构建支持与反对理由
- 生成 IRAC 论证文本
- 做基础验证与风险提示

## 三、目前已完成的工作

### 1. 总体架构与目录骨架已搭建

目前已经在新目录 `TaoxinAI/` 下完成独立重构骨架，包括：

- `api/`
- `pipelines/`
- `schemas/`
- `services/`
- `repositories/`
- `data_import/`
- `templates/`
- `tests/`

该目录不破坏原有旧项目原型，便于在原型基础上独立推进新架构。

### 2. 后端接口骨架已建立

目前已经建立新的 FastAPI 入口与三个核心接口骨架：

- `/api/chat`
- `/api/analyze`
- `/api/generate-document`

其中：

- `chat` 用于多轮收集案件信息
- `analyze` 用于执行完整分析
- `generate-document` 用于生成文书草稿

### 3. 中间数据结构已统一

已经完成一版统一 schema 设计，包括：

- `CaseInput`
- `RetrievedAuthority`
- `Issue`
- `FactItem`
- `ElementMatch`
- `ReasonBundle`
- `IRACDraft`
- `VerificationResult`

这为后续多人协作提供了统一的数据契约，降低了模块之间的冲突风险。

### 4. 讨薪场景主流程已形成第一版闭环

当前已经完成一版基础可运行的主流程：

- 输入标准化
- 候选依据召回
- 规则版争点识别
- 规则版事实抽取
- 基础要件匹配
- 理由构建
- IRAC 草稿生成
- 基础完整性检查

虽然当前仍是骨架版本，但系统已经不再只是“检索后直接回答”，而是开始具备分阶段法律分析链条。

### 5. 检索与模板部分已开始落地

目前已完成：

- 检索服务目录搭建
- Chroma repository 封装
- 本地模板目录和规则模板目录搭建
- 模板文件与文书模板初始化
- 数据导入脚本初版

并且新主线已开始复用旧 `huxin_backend` 中已有的知识库与资料。

### 6. 测试骨架已建立

目前已经补充基础测试文件，并将原先纯占位测试改为有实际断言的最小测试，包括：

- 输入预处理测试
- 争点识别测试
- 事实抽取测试
- 要件匹配测试
- IRAC 生成测试
- Pipeline 链路测试

## 四、分工进展情况

### 1. 主线部分

主线部分已完成：

- 新目录重构
- API 骨架
- Pipeline 主流程
- Schema 统一
- 规则版分析主链

### 2. 检索与模板部分

检索、模板、导入部分已经有实际实现，不再只是空文件，主要完成了：

- 导入脚本初版
- repository 初版
- retrieval 服务初版
- 模板文件初始化
- 文书模板初始化

但仍存在部分接口错配与待继续完善的问题，已经在本轮中进行了主线兼容修正。

### 3. 前端部分

前端当前仍以原型交互为主，后续将继续推进：

- 中间结果展示
- 风险提示展示
- 文书预览与导出
- 与新接口联调

## 五、目前存在的问题

虽然项目已经进入结构化重构阶段，但距离完整系统仍有差距，当前主要问题包括：

- 检索层的结构化元数据仍不够完善
- 模板和规则库内容仍偏少
- 当前事实抽取和要件匹配仍以规则骨架为主，精细度有限
- 验证层目前只有基础完整性检查，尚未接入更强的引用核验与 NLI
- 前端尚未完全切换到新主线接口
- 测试还需要继续扩充为真实样例集

## 六、当前阶段成果判断

从工程角度看，项目目前已经完成了“从单轮原型向分阶段系统迁移”的关键第一步，具体体现在：

- 不再只依赖单文件后端
- 已经具备清晰的模块边界
- 已经开始形成完整分析链条
- 已经为多人并行开发搭好基础骨架

因此，中期阶段可以认为项目已经完成了：

- 架构设计阶段
- 骨架实现阶段
- 初步功能联通阶段

## 八、结论

截至中期节点，项目已完成从“普通法律问答原型”向“讨薪方向分阶段法律论证系统”转型的基础工作，当前已具备继续深挖和展示的良好工程基础。

后续重点将从“骨架搭建”转向“内容完善、验证增强与联调展示”。
